## 基本原则

### 1. 清晰明确 (Be Specific)

**核心**: 使用具体的语言，避免模糊和歧义。

| ❌ 不好 | ✅ 好 |
|--------|-------|
| "优化代码" | "优化 python 脚本的性能，将查询时间从 3s 降至 500ms" |
| "写一个函数" | "写一个 Python 函数，使用 argparse 解析 --table 参数，返回 JSON 格式的 SAP 表数据" |
| "添加认证" | "使用 JWT Bearer Token 认证，在请求头中解析 Authorization: Bearer <token>" |

**理由**: AI 模型对具体指令的响应更精准，模糊指令导致模糊结果。

---

### 2. 包含上下文 (Provide Context)

**核心**: 说明背景、约束和现有模式。

```markdown
# 好的提示词示例

我正在开发一个 Laravel REST API 项目，使用以下配置：
- PHP 8.3+, Laravel 11
- MySQL 8.0
- 已有代码风格遵循 PSR-12

请创建一个 UserController，需要：
1. 包含 CRUD 方法
2. 返回 JSON 响应，格式为 {success: bool, data: mixed, message: string}
3. 使用 Laravel Form Request 验证
```

**理由**: 上下文帮助 AI 理解你的技术栈和约束，避免不兼容的实现。

---

### 3. 指定输出格式 (Specify Output Format)

**核心**: 明确告诉 AI 你期望的输出结构。

| ❌ 不好 | ✅ 好 |
|--------|-------|
| "输出代码" | "输出完整的 .py 文件，包含类型注解和 docstring" |
| "写个文档" | "输出 Markdown 格式文档，包含：1) 安装步骤 2) API 使用示例 3) 配置说明" |
| "给一些建议" | "以 JSON 格式输出建议：{issue: string, severity: 'high'|'medium'|'low', fix: string}" |

**理由**: 减少来回修正，提高一次性成功率。

---

### 4. 分解复杂任务 (Decompose Complex Tasks)

**核心**: 将大任务拆分为可管理的小步骤。

```markdown
# ❌ 复杂任务
"帮我做一个完整的用户管理系统，包括注册、登录、权限、API"

# ✅ 分解后的任务
1. "创建 User 模型，包含 id, name, email, password, role 字段"
2. "创建 UserController，包含 index, show, store 方法"
3. "实现 JWT 认证中间件"
4. "添加基于角色的权限控制中间件"
```

**理由**: 分解后每个任务更易理解、执行和验证。

## 好 vs 坏示例对比

### 示例 1: 功能请求

#### ❌ 差提示词 (低质量)
```
写一个获取 SAP 表数据的脚本
```

**问题**:
- 未说明使用什么语言
- 未说明调用什么 API
- 未说明输入输出格式
- 未说明错误处理要求

#### ✅ 好提示词 (高质量)
```
使用 Python 3.10+ 和 requests 库，写一个脚本：
1. 通过 ADT REST API 获取 SAP 表数据
2. 端点格式: http://{host}:{port}/sap/bc/adt/ddic/tables/{table}/data?$top={n}
3. 使用 HTTP Basic Auth (u/p 在配置文件 sap_config.yaml 中)
4. 输出 JSON 格式: {"success": bool, "data": [], "count": int}
5. 添加超时重试机制 (3次, 指数退避)
6. 添加详细日志记录
```

**改进点**:
- 明确语言和依赖
- 指定 API 端点格式
- 说明认证方式
- 定义输出格式
- 包含错误处理策略

### 示例 2: 技能定义

#### ❌ 差 SKILL.md
```yaml
---
name: my-skill
description: 做事情
---
# 我的技能
帮助用户做各种事情
```

**问题**:
- name 包含下划线，应使用连字符
- description 太笼统
- 缺少触发条件说明

#### ✅ 好 SKILL.md
```yaml
---
name: sap-table-fetcher
description: 获取 SAP 表结构和数据。使用场景：(1) 用户需要获取表数据 (2) 用户需要获取表结构 (3) 用户需要分析 SAP 表关系
---

# SAP Table Fetcher

在用户请求获取 SAP 系统中的表数据或表结构时调用此技能。

## 触发条件
- 用户说 "获取 XXX 表数据"
- 用户说 "查看 XXX 表结构"
- 用户说 "SAP 表分析"

## 执行步骤

1. **确认表名**
   - 解析用户输入的表名（如 MARA, ZTJQ0001）
   - 验证表名格式

2. **调用 API**
   - 表结构: `/sap/bc/adt/dictionary/tables/{表名}/definition`
   - 表数据: `/sap/bc/adt/ddic/tables/{表名}/data?$top=10`
   - 携带 Basic Auth 认证

3. **解析响应**
   - 结构: 提取字段名、类型、描述
   - 数据: 转换为 JSON 数组

4. **格式化输出**
   ```json
   {
     "table": "MARA",
     "fields": [...],
     "data": [...],
     "count": 10
   }
   ```

## 配置
- 配置文件: SAP_CONFIG (USER, PASSWD, CLIENT, HOST, PORT)
- 使用项目现有配置，不要硬编码
```

**改进点**:
- name 使用小写字母和连字符
- description 明确包含触发条件
- 添加了详细的执行步骤
- 说明输入输出格式
- 引用项目配置

## SKILL.md 最佳实践

### 1. Frontmatter 规范

```yaml
---
name: skill-name          # 必须: 1-64字符, 小写字母+单连字符
description: "描述文字"   # 必须: 1-1024字符, 清晰说明触发条件
license: MIT              # 可选: MIT, Apache等
compatibility: opencode  # 可选: opencode, claude, agents
metadata:                 # 可选: 扩展元数据
  audience: developers
  workflow: sap
---
```

### 2. 结构化内容

```markdown
## What I do
- 简洁描述技能功能 (1-2句话)

## When to use me
- 明确触发场景 (bullet points)
- 包含示例触发语

## How I work
- 详细步骤说明
- 输入输出格式
- 依赖和配置

## Example
- 提供使用示例
```

### 3. 描述编写技巧

**技巧 1: 包含触发关键词**
```yaml
description: "获取 SAP 表结构和数据。使用场景：(1) 获取表数据 (2) 获取表结构 (3) 分析 SAP 表"
```

**技巧 2: 使用 "Use when" 模式**
```yaml
description: "Create documentation optimized for AI agent consumption. Use when writing SKILL.md files, README files, or any documentation that will be read by LLMs."
```

## Agent 提示词技巧

### 1. 使用 System Prompt 定义角色

```python
# ❌ 无角色定义
"写一个 API"

# ✅ 有角色定义
"""
你是一个专业的 SAP 集成工程师，专长于 ABAP 系统 REST API 开发。
请帮我创建一个 FastAPI 端点，用于：
1. 接收 table_name 参数
2. 调用 SAP ADT API 获取表数据
3. 返回标准 JSON 响应
4. 添加错误处理和日志
"""
```

### 2. 提供示例 (Few-shot)

```python
# ❌ 无示例
"写一个验证邮箱的函数"

# ✅ 有示例
"""
写一个验证邮箱的函数，要求：
1. 使用正则表达式验证格式
2. 返回 True/False

示例：
validate_email("test@example.com") -> True
validate_email("invalid") -> False
validate_email("user@domain") -> False
"""
```

### 3. 指定约束条件

```python
# ❌ 无约束
"优化这段代码"

# ✅ 有约束
"""
优化以下 Python 代码，要求：
1. 时间复杂度从 O(n²) 降至 O(n)
2. 保持原有函数签名不变
3. 添加类型注解
4. 添加单元测试
5. 不要使用第三方库
"""
```

## 常见模式

### 模式 1: 任务分解模式

```
目标: [最终目标]
当前状态: [现有条件]
约束: [限制条件]

请按以下步骤执行：
1. [步骤1]
2. [步骤2]
3. [步骤3]
```

### 模式 2: 角色扮演模式

```
你是一个 [角色]，专长于 [领域]。
[背景描述]

请帮我 [任务]，需要满足：
1. [要求1]
2. [要求2]
```

### 模式 3: 迭代优化模式

```
初始代码: [代码片段]
问题: [具体问题]
目标: [优化目标]

请提供：
1. 改进方案
2. 改进后的代码
3. 性能对比 (改进前 vs 改进后)
```

### 模式 4: 文档生成模式

```
请为 [代码/项目] 生成文档，包含：
1. 概述
2. 安装/配置
3. 使用示例
4. API 参考
5. 常见问题

格式: Markdown
输出: 直接输出文档内容
```

## 理由说明

### 为什么清晰明确很重要？

1. **减少歧义**: AI 模型会猜测你的意图，导致不符合预期
2. **提高效率**: 减少来回修正的次数
3. **可重复性**: 明确的提示词可以重复使用

### 为什么需要上下文？

1. **技术匹配**: AI 需要知道你的技术栈才能给出兼容的方案
2. **风格一致**: 代码风格、命名规范需要参考现有代码
3. **避免冲突**: 避免引入与现有系统不兼容的方案

### 为什么分解任务？

1. **降低复杂度**: 每个小任务更容易理解和执行
2. **便于验证**: 可以逐步检查每一步的结果
3. **错误定位**: 问题出现时容易定位到具体步骤

### 为什么指定输出格式？

1. **减少后处理**: 直接得到期望的格式，无需手动转换
2. **明确期望**: 让 AI 知道"好"的标准是什么
3. **自动化友好**: 格式化的输出更易于程序处理

## 总结检查清单

编写提示词时，检查以下各项：

- [ ] **目标明确**: 是否清楚地说明了要做什么？
- [ ] **上下文完整**: 是否提供了技术栈、约束、现有模式？
- [ ] **输出格式**: 是否指定了期望的输出格式？
- [ ] **复杂度适当**: 是否将复杂任务分解为步骤？
- [ ] **示例提供**: 是否包含示例或参考？
- [ ] **约束明确**: 是否说明了限制条件？
